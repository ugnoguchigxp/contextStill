# 画像生成モデルを活用した Web / UI デザインのベストプラクティス

## 目的

このドキュメントは、画像生成・画像編集モデルを Web サイト、LP、アプリケーション画面、UI モックアップのデザイン検討に活用する際のベストプラクティスをまとめたものです。

対象とするモデルは特定のサービスに限定しません。  
OpenAI の画像生成モデル、Google Gemini 系の画像生成・編集モデル、Nano Banana のような画像編集に強いモデル、その他のマルチモーダル画像生成モデルにも共通して適用できる考え方を扱います。

画像生成モデルは、ビジュアル方向性、構図、配色、雰囲気、UIパターンの探索に有効です。一方で、生成画像はそのまま実装仕様にはなりません。実際のWebページやアプリケーションとして利用するには、画像を構造化し、アクセシブルでレスポンシブかつ保守可能なUI設計へ翻訳する必要があります。

---

## 基本方針

画像生成モデルは、完成済みの設計図ではなく、**視覚的な方向性を探索するための参考資料**として扱う。

生成画像をそのままピクセル単位で再現するのではなく、以下の要素に分解して活用する。

- 画面の目的
- 情報の優先順位
- レイアウト構造
- 主要コンポーネント
- 余白のリズム
- 配色
- タイポグラフィ
- 装飾の方向性
- レスポンシブ時の構成
- アクセシビリティ上の懸念
- 実装難易度
- 保守性

---

## 1. 生成画像は「完成デザイン」ではなく「視覚参考資料」として扱う

画像生成モデルは、見栄えの良いビジュアルを短時間で作ることに優れている。一方で、生成画像には以下のような不確実性がある。

- 余白やサイズが厳密ではない
- テキストが正確とは限らない
- UI構造が実装可能とは限らない
- レスポンシブ時の挙動が定義されていない
- アクセシビリティが保証されていない
- 既存ブランドや既存UIに似た表現が混ざる可能性がある
- 画像としては成立しても、Web実装としては重すぎる場合がある

したがって、生成画像は「そのまま作るもの」ではなく、「方向性を読み取り、実装可能な仕様へ変換するもの」として扱う。

---

## 2. モデル固有ではなく、共通の実装可能性ガードレールを使う

利用する画像生成モデルが OpenAI 系であっても、Gemini 系であっても、その他のモデルであっても、Web / UI デザイン用途では同じ制約を入れるべきである。

重要なのは、モデル名ではなく、**生成結果が実装可能なUIとして成立すること**である。

### 推奨プロンプト

```text
この画像は、実際のレスポンシブWebページとしてHTML/CSS/最小限のJavaScriptで実装する前提です。
標準的なHTML、CSS、SVGで再現できるレイアウトと視覚効果に限定してください。
過度な3D、複雑な粒子表現、流体表現、強すぎるガラス歪み、実装困難な影、読めない小さい文字、保守不能な装飾は避けてください。
明確なレイアウト、読みやすい文字、十分なコントラスト、再利用可能なコンポーネント、レスポンシブ対応を優先してください。
```

### 英語版

```text
This image is intended to be implemented as a real responsive web page.
Use only layouts and visual effects that can be reproduced with standard HTML, CSS, SVG, and minimal JavaScript.
Avoid unrealistic 3D, complex particle effects, heavy glass distortion, fluid simulations, impossible shadows, unreadable micro text, and decorative elements that would not translate into maintainable frontend code.
Prioritize clear layout, readable typography, accessible contrast, reusable components, and responsive behavior.
```

---

## 3. 利用モデルを切り替えても変わらない判断基準を持つ

画像生成モデルごとに得意不得意は異なる。

例として、あるモデルは構図や質感に強く、別のモデルは画像編集や一貫性維持に強く、また別のモデルはテキスト入りUIや図解に強い場合がある。

しかし、Web / UI デザイン活用における評価基準は共通である。

- 実装可能か
- 読みやすいか
- レスポンシブにできるか
- アクセシブルか
- 情報階層が明確か
- 既存ブランドに似すぎていないか
- 保守可能なコンポーネントに分解できるか
- 画像の美しさではなく、UIとして使えるか

モデルの違いは「生成手段」の違いであり、最終的な判断基準は「実用的なWeb/UIとして成立するか」で統一する。

---

## 4. ページ全体ではなく、コンポーネント単位で生成する

大きなページ全体を1枚の画像で生成すると、細部が曖昧になりやすく、実装時に分解しにくい。

以下のように、UIを小さな単位に分けて生成する方が扱いやすい。

- Hero セクション
- Feature カード
- Pricing セクション
- CTA ブロック
- Dashboard モック
- Form UI
- Data Table
- Empty State
- Settings 画面
- Mobile Navigation
- Footer

特にWebアプリケーションでは、以下のように構造ごとに分けると実装しやすい。

- Layout Shell
- Navigation
- Content Card
- Form
- Table
- Modal
- Notification
- Loading / Error / Empty State

---

## 5. 画像内テキストは最小限にする

画像内に本番文言を焼き込むことは避ける。

理由は以下の通り。

- SEOに効かない
- スクリーンリーダーで読めない
- 翻訳や修正が難しい
- レスポンシブで崩れやすい
- 文字化けや誤字が起きる可能性がある
- HTMLテキストとして管理できない

画像生成では、見出し・本文・CTA・ラベルなどの「位置」や「階層」だけを確認し、実際のテキストはHTML側で実装する。

### 推奨プロンプト

```text
画像内に本番用の文言を入れないでください。
必要な場合は、情報階層を示すための短いプレースホルダーだけを使ってください。
実際の見出し、本文、ボタン文言、フォームラベル、価格、注釈はHTMLテキストとして実装します。
```

### 英語版

```text
Do not render final production copy inside the image.
Use simple placeholder text only where needed to show hierarchy.
All real headings, body text, button labels, form labels, prices, and notes will be implemented as HTML text.
```

## 6. ピクセル模写ではなく、構造へ翻訳する

生成画像をHTML/CSSに変換する際は、ピクセル単位で再現しようとしない。

重視すべきなのは、以下の再現である。

- デザイン意図
- 情報階層
- 視線誘導
- 余白のリズム
- コンポーネント構造
- トーン
- 操作しやすさ
- レスポンシブ対応

避けるべき実装は以下。

- 絶対位置だらけのレイアウト
- 固定幅依存
- 画像をフルページ背景として使う実装
- 画像内テキストの流用
- DOM構造が意味を持たない実装
- hover / focus / keyboard 操作を考慮しない実装
- モバイルで破綻する実装

推奨する実装方針は以下。

- セマンティックHTMLを使う
- CSS Grid / Flexbox を使う
- 色・余白・角丸・影をデザイントークン化する
- テキストはHTMLとして実装する
- 画像は装飾または補助要素として扱う
- モバイルファーストで設計する
- 再利用可能なコンポーネントに分ける

---

## 7. レスポンシブ設計を前提にする

画像生成時点で、PC表示だけではなくモバイル表示も想定する。

### 指定すべき内容

- Desktop では何カラムか
- Tablet ではどう折り返すか
- Mobile ではどの順序で縦積みするか
- CTAはどこに残すか
- ナビゲーションはどう変化するか
- カードや表はどう表示するか
- 画像・イラスト・背景装飾はモバイルで省略できるか

### 推奨プロンプト

```text
レスポンシブ対応を前提にしてください。
デスクトップでは12カラムグリッドを想定し、モバイルでは1カラムに自然に縦積みできる構成にしてください。
重要なCTA、見出し、本文、フォームはモバイルでも見やすく、操作しやすい配置にしてください。
```

---

## 8. アクセシビリティを最初から考慮する

画像として美しくても、Webページとして使いにくければ実用性は低い。

画像生成段階から、以下を制約として入れる。

- 文字と背景のコントラストを十分に確保する
- 本文は16px相当以上を想定する
- 小さすぎる文字を避ける
- 複雑な背景の上に本文を置かない
- CTAや入力欄を十分なサイズにする
- 色だけで意味を伝えない
- 操作可能な要素を視覚的に判別しやすくする
- hover / focus 状態を後から設計できる構造にする

### 推奨プロンプト

```text
アクセシビリティを重視してください。
文字と背景のコントラストを十分に確保し、本文は16px相当以上で読める前提にしてください。
複雑な背景の上に重要なテキストを置かず、ボタンやフォームなどの操作要素は明確に識別できるようにしてください。
色だけで状態や重要度を表現しないでください。
```

---

## 9. 実装困難な表現は置き換える

画像上では魅力的でも、実装すると重い・壊れやすい・保守しにくい表現がある。

以下のような置換を検討する。

| 画像上の表現 | 実装時の代替 |
|---|---|
| 複雑な3D背景 | CSSグラデーション + SVG装飾 |
| 粒子・煙・流体 | 静的なぼかし円、軽い背景画像 |
| 強いガラス歪み | 控えめな `backdrop-filter` |
| 複雑な影 | 2〜3段階の shadow token |
| 不規則なカード配置 | Grid / Masonry風レイアウトに簡略化 |
| 細かすぎる文字 | HTMLテキストに置換 |
| 装飾的なアイコン群 | SVGアイコンセットに置換 |
| 写真的なUIモック | 実装可能なカードUIに再構成 |

### 判断基準

以下のいずれかに該当する表現は、そのまま実装しない。

- パフォーマンスを大きく落とす
- モバイルで破綻する
- アクセシビリティを損なう
- 実装が過度に複雑になる
- 保守が難しくなる
- 意味より装飾が勝っている
- 実装後の更新が難しい

---

## 10. 既存ブランド・既存UIの模倣を避ける

画像生成時に、有名サービスや既存ブランドに似た表現が混ざることがある。

以下は避ける。

- 既存企業のロゴに似たロゴ
- 有名サービスの画面に似すぎたUI
- 商標やブランドカラーの強い模倣
- 既存アイコンセットの模倣
- 「Apple風」「Stripe風」「Notion風」など、特定ブランドを想起させる指定

### 推奨プロンプト

```text
オリジナルのデザインにしてください。
既存ブランド、既存サービスのUI、ロゴ、商標、アイコンセット、ブランドビジュアルに似せないでください。
一般的で洗練された現代的なWebデザインとして作成してください。
```

### 英語版

```text
Create an original design.
Do not imitate or closely resemble any existing brand, product UI, logo, icon set, or trademarked visual identity.
Use a generic but polished modern web design style.
```

---

## 11. 反復編集では「変更するもの」と「維持するもの」を明示する

画像を再生成・編集する際は、変更点だけでなく、維持する点も明示する。

曖昧な指示では、意図しない部分まで変わる可能性がある。これは画像編集に強いモデルを使う場合でも同じである。

### 推奨フォーマット

```text
変更するもの:
- CTAボタンの形状
- ボタン色
- Hero右側のカード表現

維持するもの:
- 全体レイアウト
- 余白
- 見出し位置
- 背景のトーン
- カードの角丸
- ナビゲーション構成
```

### 英語版

```text
Change only:
- CTA button shape
- Button color
- Product preview card style

Keep unchanged:
- Overall layout
- Spacing rhythm
- Heading position
- Background tone
- Card border radius
- Navigation structure
```

## 12. モデル選定時の観点

特定のモデル名に依存せず、用途ごとに適したモデルを選ぶ。

### 新規デザイン案の探索

重視する点:

- 構図の多様性
- ビジュアル品質
- スタイル制御
- 複数案の出しやすさ
- 指示追従性

向いている用途:

- LPの方向性検討
- Hero案の探索
- ムードボード作成
- 配色・質感の比較

### 既存画像の編集

重視する点:

- 元画像の構図維持
- 部分編集の安定性
- キャラクター・UI・物体の一貫性
- 変更箇所以外を保持する能力

向いている用途:

- 既存UIモックの調整
- ボタンやカードの見た目変更
- 背景や色味の変更
- 画像内の一部要素の差し替え

### UIモックアップ

重視する点:

- レイアウトの整合性
- 読みやすさ
- 文字領域の扱い
- UI部品の自然さ
- コンポーネント境界の明確さ

向いている用途:

- Webアプリ画面
- ダッシュボード
- フォーム画面
- 設定画面
- 管理画面

### 図解・説明画像

重視する点:

- 構造の明確さ
- ラベルの扱いやすさ
- アイコンや矢印の整合性
- 情報量の制御

向いている用途:

- サービス説明図
- ワークフロー図
- 概念図
- アーキテクチャのラフ案

## 13. モデルをまたいで使える共通プロンプト構造

どの画像生成モデルでも、プロンプトは以下の構造にすると安定しやすい。

```text
目的:
[この画像で検討したいこと]

対象:
[LP / Hero / Dashboard / Form / Pricing / Mobile UI など]

利用前提:
[Web実装、資料用、デザイン検討用、既存画像の編集など]

構成:
[必要なセクションやUI要素]

レイアウト:
[カラム数、グリッド、配置、レスポンシブ条件]

ビジュアルトーン:
[clean, calm, professional, playful など]

制約:
[実装可能性、アクセシビリティ、画像内テキスト最小化、ブランド模倣禁止など]

禁止事項:
[過度な3D、読めない文字、既存ブランド類似、複雑すぎる背景など]

出力の使い道:
[実装参考、方向性確認、既存UI改善、クライアント確認用など]
```

## 14. 生成後に実装前レビューを行う

生成画像を見たら、すぐにコード化せず、まずレビューを行う。

### レビュー項目

1. 画面の目的は明確か
2. 情報階層は分かりやすいか
3. 主要コンポーネントに分解できるか
4. Grid / Flexbox で実装できるか
5. モバイルで自然に並び替えられるか
6. 画像内テキストをHTML化できるか
7. 背景や装飾が邪魔になっていないか
8. コントラストは十分か
9. CTAは目立つか
10. 実装困難な視覚効果はないか
11. 既存ブランドに似すぎていないか
12. 削った方がよい要素はないか

### 実装前レビュー用プロンプト

```text
この画像デザインを、実際のWebページとして実装する前提でレビューしてください。

以下を出力してください。

1. ページの目的
2. コンポーネント分解
3. レイアウト方針
4. レスポンシブ時の挙動
5. HTMLとして実装すべきテキスト
6. 簡略化すべき装飾
7. アクセシビリティ上の懸念
8. 実装リスク
9. CSS方針
10. コード化前に修正すべき点
```

## 15. デザイントークンへ変換する

画像から直接CSSを書くのではなく、まずデザイントークンへ変換する。

### 例

```text
Colors:
- background: #FFFFFF
- surface: #F8FAFC
- primary: #2563EB
- text-primary: #0F172A
- text-secondary: #475569
- border: #E2E8F0

Spacing:
- xs: 4px
- sm: 8px
- md: 16px
- lg: 24px
- xl: 40px
- 2xl: 64px

Radius:
- sm: 6px
- md: 12px
- lg: 20px

Shadow:
- card: subtle shadow
- elevated: medium shadow
```

画像から読み取った印象を、コードで再利用できる抽象値に置き換えることで、保守性が高くなる。

## 16. 推奨ワークフロー

### Phase 1: 要件整理

- 目的
- ターゲット
- ページ種別
- 必要セクション
- ブランドトーン
- 技術制約
- 禁止事項

### Phase 2: モデル選定

- 新規生成に向くモデルか
- 既存画像編集に向くモデルか
- UIモックに向くモデルか
- 反復編集に向くモデルか
- コストや速度の制約に合うか

### Phase 3: 画像生成・編集

- セクション単位で生成
- 実装可能性を制約に入れる
- レスポンシブを前提にする
- 本番テキストを画像に入れない
- 既存ブランドの模倣を避ける
- 編集時は変更点と維持点を明示する

### Phase 4: 画像レビュー

- 実装可能性
- 情報階層
- アクセシビリティ
- レスポンシブ
- 実装困難な表現
- 削るべき装飾

### Phase 5: 実装仕様化

- コンポーネント一覧
- DOM構造
- CSS方針
- デザイントークン
- ブレイクポイント
- 画像として残す要素
- HTML化するテキスト

### Phase 6: コード化

- セマンティックHTML
- CSS Grid / Flexbox
- 再利用可能なコンポーネント
- 実テキスト
- アクセシブルな操作要素
- モバイルファースト

### Phase 7: 実装後チェック

- モバイルで崩れないか
- キーボード操作できるか
- コントラストは十分か
- 固定幅に依存していないか
- 画像スクリーンショット依存になっていないか
- 余計な装飾が性能を落としていないか

## 23. まとめ

画像生成モデルは、Web/UIデザインにおける視覚的な探索、方向性の確認、構図やトーンの検討に非常に有用である。

ただし、どのモデルを使う場合でも、生成画像は実装仕様そのものではない。  
実用的なWeb/UIにするためには、生成画像を以下のように扱う必要がある。

```text
画像生成・編集 = 視覚的な方向性探索
画像レビュー = 実装可能性とUXの検査
仕様化 = コンポーネント、レイアウト、デザイントークンへの変換
コード化 = アクセシブルでレスポンシブな実装
```

最も重要な原則は以下である。

```text
画像生成モデルは視覚的な方向性探索に使い、実装仕様そのものとは見なさない。
生成画像を、アクセシブルでレスポンシブかつ保守可能なWeb/UIコンポーネントへ翻訳してからコード化する。
```

英語では次のように表現できる。

```text
Use image generation models to explore visual intent, not to define implementation truth.
Then translate the image into accessible, responsive, maintainable web and UI components.
```
